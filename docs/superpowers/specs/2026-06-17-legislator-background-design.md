# 公職人員背景資料庫 — 設計規格

- 日期：2026-06-17
- 狀態：設計定稿（待 review）
- 對標：legislator-wealth.tw（設計水準與公信力）

## 1. 目標與定位

以**可信、可查證**的方式，公開現任公職人員的**經歷、司法判決、爭議事件與財產申報**，讓選民能在一處看到完整背景。

核心差異化：legislator-wealth.tw 只做財產；本站擴及「背景・經歷・司法・爭議」，但同樣堅持每筆資料皆有合法公開來源。

**最高原則：可靠性。** 寧可少寫、不寫沒出處的東西。整個系統的設計都服從這一點。

## 2. 範圍

### MVP（第一階段）
- **對象**：現任立委（第 11 屆，113 位）+ 現任 22 縣市首長。
- **作法**：先以 **5–10 位真實人物**做精選原型，驗證資料呈現與法律標註方式，再投入大規模資料蒐集。
- **資料取得**：原型階段純人工策展與查核。

### 第二階段
- 寫爬蟲抓取結構化公開資料（裁判書系統、立法院、中選會），媒體爭議仍人工補。

### 後續擴充（schema 需預留，但不在此規格實作）
- 縣市議員（全台約 900+ 位）。
- 資料模型以通用「公職人員」設計，加入議員不需改結構。

### 明確不做（YAGNI）
- 帳號／登入系統
- 留言／討論區
- 即時通知
- 歷屆全集（僅現任）
- 群眾投稿（原型階段不開放，避免成為謠言溫床；未來若開放需審核機制）
- 對人物的價值判斷或評分標籤

## 3. 資料定義與來源（關鍵）

| 資料類別 | 合法公開來源 | 標註要求 |
|---|---|---|
| 背景・經歷 | 立法院官方、中選會候選人學經歷申報 | 附來源 |
| 司法判決 | 司法院裁判書查詢系統 | **判決結果 + 是否定讞**；常駐無罪推定聲明 |
| 媒體爭議 | 新聞報導、事實查核 | **目前狀態 + 報導日期**；可多筆來源 |
| 財產申報 | 監察院公職人員財產申報公報 | 附年度與公報來源 |

「前科」一詞在台灣指涉個人刑事紀錄，屬受個資法保護、不可彙整公開之資料。**本站不使用「前科」概念**，改以兩個分開、合法公開的類別呈現：**司法判決紀錄**與**媒體爭議事件**，且明確區分、各自標示狀態與出處。

## 4. 資料模型（PostgreSQL / Supabase）

通用「公職人員」設計。所有事實型資料強制關聯來源。

### `officials`
- `id`、`name`、`party`、`office_type`（enum: `legislator` / `mayor_magistrate` / `councilor`）
- `district`（選區/轄區）、`term`（現任屆期）、`photo_url`、`bio`（簡短介紹）
- `is_incumbent`

### `careers`（背景・經歷）
- `id`、`official_id` (FK)、`title`、`organization`、`start_date`、`end_date`(nullable)、`source_id` (FK)

### `judgments`（司法判決）
- `id`、`official_id` (FK)、`case_reason`（案由）、`court`、`case_number`（字號）
- `outcome`（判決結果文字）、`is_final`（是否定讞 bool）、`judgment_date`
- `judgment_url`（裁判書連結）、`source_id` (FK)

### `controversies`（媒體爭議）
- `id`、`official_id` (FK)、`title`、`summary`、`status`（enum: `investigating` / `indicted` / `first_instance` / `settled` / `cleared` / `other`）
- `event_date`、`report_date`
- 來源為多筆：透過 `controversy_sources` 關聯 `sources`

### `asset_declarations`（財產申報）
- `id`、`official_id` (FK)、`year`、`total_amount`、`source_id` (FK)

### `sources`（出處中心）
- `id`、`url`、`type`（enum: `court` / `news` / `gov` / `gazette` / `factcheck`）、`title`、`retrieved_at`

### 完整性約束
- `judgments`、`asset_declarations`、`careers`：`source_id` 為 NOT NULL。
- `controversies`：至少一筆 `controversy_sources`（以 DB 約束或寫入層 / 審核流程保證）。
- **無出處的事實型資料不得上架。**

## 5. 可靠性與法律機制（貫穿全站）

1. **無出處不上架**：DB 層 NOT NULL / 關聯約束強制每筆事實資料有來源。
2. **判決 ≠ 定罪**：判決卡片固定顯示「結果 + 是否定讞」；檔案頁與關於頁常駐「無罪推定」說明。
3. **爭議標狀態與日期**：偵查中／起訴／一審…／已和解／查無不法，並標報導日期，避免拿舊聞當現況。
4. **中立用語**：只陳述事實與出處，不下價值判斷或情緒性標籤。
5. **更正管道**：頁尾提供更正／申訴聯絡方式；關於頁說明方法論。

## 6. 系統架構

### 前台：Astro（SSG）
- 逐人檔案頁、總覽頁、關於頁皆於建置時靜態產生 → 載入快、SEO 最佳。
- 「篩選排序列表」為唯一互動需求：用一個互動 island 實作，資料由建置時產出的 JSON 餵入，**前台不直連資料庫**。
- 單一強調色、編輯風排版、支援 dark mode（見視覺）。

### 資料／後台：Supabase
- PostgreSQL 儲存上述模型。
- RLS：公開資料**唯讀**對外開放；寫入僅限後台角色。
- 策展／審核：使用 Supabase 內建後台或簡易 admin 介面，人工輸入與審核。
- 建置流程：build 時從 Supabase 拉資料 → 產生靜態頁與列表 JSON。

### 爬蟲（第二階段）
- 抓裁判書系統、立法院、中選會結構化資料 → 寫入 Supabase。
- 媒體爭議維持人工補錄與查核。

## 7. 頁面與功能

### 首頁／總覽
- 可排序、可篩選的資料表格：政黨、職位類型、選區、判決數、爭議數、財產申報額。
- 關鍵字搜尋（姓名）。
- 互動 island，資料來自靜態 JSON。

### 逐人檔案頁
- 區塊順序：**經歷時間軸 → 司法判決 → 爭議事件 → 財產申報**。
- 每筆資料附可點擊出處。
- 判決區塊常駐無罪推定說明。

### 關於頁
- 資料來源清單、方法論、無罪推定聲明、更正管道、不做價值判斷之立場說明。

## 8. 視覺設計

- **基調**：「公民資料庫」風（淺色、中性克制，像 g0v／維基），避免爆料八卦觀感。
- **列表**：高密度可排序表格（取 legislator-wealth 的資訊密度）。
- **手法（刻意去 AI 味、走編輯設計感）**：
  - 襯線標題（Noto Serif TC）+ 無襯線內文做對比。
  - **單一強調色（朱紅 ~#b3271e / dark ~#e4574c）**；只有「有資料」的數字上色，`0` 淡化。
  - 數字用 tabular 等寬；**細線分隔取代色塊**。
  - 支援 dark mode（同一套邏輯換深底）。
- 實作階段以 UI/UX skill 把關，避免通用 AI 色塊風。

## 9. 測試策略

- **資料完整性**：測試確保事實型資料無來源時無法上架（DB 約束 / 寫入層驗證）。
- **建置產出**：測試 build 能正確由 Supabase 產出靜態頁與列表 JSON。
- **列表互動**：測試篩選／排序／搜尋邏輯（對 JSON 資料）。
- **法律標註**：測試判決卡片必含「結果 + 是否定讞」、爭議必含狀態與日期。
